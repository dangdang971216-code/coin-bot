# 코인봇 인수인계 v2.6.25

## 0. 최신 파일
- 코드 파일: `수익형_v2.6.25.py`
- GitHub 자동/즉시 배포 지정 파일: `DEPLOY_TARGET.txt`
- DEPLOY_TARGET.txt 내용: `수익형_v2.6.25.py`
- 이번 버전은 전략 조건/매수 문턱을 바꾸지 않았다.
- 이번 버전의 목적은 SSH 의존 줄이기, 출력직원 다이어트, 상태판 오류 판단 개선이다.

---

## 1. 이번 v2.6.25에서 한 것

### 1) 텔레그램 즉시 업데이트 명령 추가
- `/update_bot` 추가.
- 서버에 이미 만들어 둔 `/usr/local/bin/tradingbot_auto_deploy.sh`를 호출한다.
- GitHub에서 최신 파일을 가져오고, `DEPLOY_TARGET.txt`에 적힌 파일명을 기준으로 반영한다.
- 문법검사 실패 시 기존 `bot.py`를 유지하는 서버 스크립트 구조를 사용한다.
- 배포 과정에서 봇이 재시작될 수 있으므로 `/update_bot`은 batch 안이 아니라 단독 실행을 권장한다.
- 배포 시작 후 결과 확인은 `/deploy_status`로 한다.

### 2) 배포 상태 확인 명령 추가
- `/deploy_status` 추가.
- 서버 스크립트 `/usr/local/bin/tradingbot_deploy_status.sh`를 호출해 최근 배포 로그, 적용 대상, 봇 상태를 확인한다.
- SSH가 잘 안 열릴 때 배포 성공/실패를 텔레그램에서 확인하기 위한 명령이다.

### 3) 서버 점검 명령 추가
- `/admin_check`, `/server_check` 추가.
- 서버 스크립트 `/usr/local/bin/tradingbot_admin_check.sh`를 호출한다.
- 메모리, 디스크, 봇 active 상태, 최근 로그를 확인한다.
- SSH가 불안정할 때 최소 서버 상태를 텔레그램에서 볼 수 있게 하기 위한 것이다.

### 4) 명령어 캐시/출력직원 다이어트
- 기존 기본 명령어 캐시가 `rank_watch`, `review_pack`, `review_dump` 같은 긴 출력을 백그라운드에서 계속 만들면서 서버 부담을 키울 수 있었다.
- v2.6.25에서는 기본 캐시를 fast-only 중심으로 바꿨다.
- 기본 캐시에서 우선 만드는 것:
  - `rank_short`
  - `review_watch`
  - `work_check`
  - `scan_health`
- 긴 출력은 필요할 때 직접 명령어로 보는 방향으로 유지한다.
- 목적은 출력직원이 매번 큰 보고서를 만들다가 명령캐시가 2~10분씩 밀리는 문제를 줄이는 것이다.

### 5) 상태판 표시 수정
- v2.6.24에서 `전체 60 → 현재가 59 → 순위 78 → 분류 40`처럼 표시되어 헷갈렸다.
- v2.6.25에서는 `거래가능 454 → 현재허브 59 → 순위 78 → 분류 40`처럼 표시를 분리했다.
- 전체 거래가능 코인 수와 현재 감시/허브에 들어온 수를 구분한다.

### 6) 현재 FAIL / 누적 FAIL 분리
- v2.6.24의 `speed_check`는 예전 write-back FAIL 누적값이 계속 보여서 지금 오류인지 과거 찌꺼기인지 헷갈렸다.
- v2.6.25는 다음을 분리한다.
  - 현재 write-back 상태
  - 누적 write-back OK/FAIL
- 판단할 때는 현재 write-back을 먼저 본다.

### 7) paper허브 정보없음 문구 보강
- `paper허브: 정보없음`만 나오면 실제 paper가 죽은 건지, 표시허브만 없는 건지 알기 어려웠다.
- v2.6.25는 가능한 경우 실제 `paper_positions`의 OPEN/PENDING 수를 같이 표시한다.
- 그래도 정확한 paper 상태는 `/paper_stats`에서 확인한다.

### 8) 표시/복기용 메모리 소프트 정리
- 전략 판단에 직접 쓰지 않는 표시용 기록 일부만 가볍게 정리한다.
- `real_flow` 최근 기록은 80개 선으로 제한한다.
- 급등복구 큐는 120개 이상 비대해지면 오래된 것을 일부 줄인다.
- 실거래/눌림/시장자료 판단 자체는 건드리지 않는다.

## 2. 이번에 건드리지 않은 것
- 눌림 재확인 라인의 매수 조건/차트 조건은 변경하지 않았다.
- 슬리피지 기준은 변경하지 않았다.
- 자동매매 허용 전략은 여전히 눌림 재확인 라인만이다.
- 1회 매수금액/동시보유/수익보호 기준은 변경하지 않았다.
- 급등복구는 삭제하지 않고, 본선/실거래를 방해하지 않는 보조직원으로 유지한다.

## 3. v2.6.25 적용 방법

### GitHub에 올릴 파일 2개
1. `수익형_v2.6.25.py`
2. `DEPLOY_TARGET.txt`

`DEPLOY_TARGET.txt` 안에는 아래 한 줄만 둔다.

```text
수익형_v2.6.25.py
```

### SSH 없이 즉시 반영
GitHub 업로드 후 텔레그램에서 단독으로 실행한다.

```text
/update_bot
```

배포 시작 메시지가 오면 20~40초 뒤 확인한다.

```text
/deploy_status
```

### 기존 SSH 반영 방식이 필요할 때
SSH가 열려 있고 직접 반영할 때는 기존 선호 형식대로 진행한다.

```bash
cd ~/trading_bot
echo "1 git 받는 중"
git fetch origin
git reset --hard origin/main
echo "2 파일 복사"
cp "수익형_v2.6.25.py" bot.py
ls -lh bot.py
echo "3 문법검사"
python3 -m py_compile bot.py && echo "문법검사 OK"
echo "4 봇 재시작"
sudo systemctl restart tradingbot && echo "재시작 명령 OK"
echo "5 상태확인"
sudo systemctl status tradingbot --no-pager
echo "6 최근로그"
journalctl -u tradingbot -n 40 --no-pager
```

## 4. v2.6.25 적용 후 확인 배치

적용 후 바로 아래를 본다.

```text
/batch
/ping
/system_check
/error_doctor
/auto_status
/real_flow
/speed_check
/deploy_status
/admin_check
```

### 정상 기준
- `/ping`: 버전 `수익형 v2.6.25`
- `/system_check`: `거래가능 454 → 현재허브 ... → 순위 ... → 분류 ...` 형태로 표시
- `/error_doctor`: 현재 write-back과 누적 FAIL이 분리되어 표시
- `/auto_status`: 자료 정상, 실보유/후보/눌림공장 상태가 짧게 표시
- `/speed_check`: 현재 write-back OK, 명령캐시가 30~90초 권역이면 좋음
- `/deploy_status`: 배포 로그 확인 가능
- `/admin_check`: 서버 상태 확인 가능

### 자동매매 켜기 전 기준
아래 조건을 확인하고 `/auto_on`을 실행한다.
- 매매자료 정상
- 현재 write-back 순위 OK
- 현재 write-back 출력 OK 또는 출력 주의가 매매판단을 막지 않는 상태
- 시장/순위/분류가 정상
- 실거래 직원 loop 정상
- `/auto_on` 안전검사에서 호가/자료 통과

## 5. 현재 큰 목표

최종 목표는 변함없다.

> 초반 선점형 + 재출발형 + 실시간 강세 초입형을 잡고, 늦은 추격형은 버리는 수익률형 봇.

현재 우선순위는 조건 조이기가 아니라 운영 안정화다.
- 서버를 당장 키우지 않고 기존 서버로 버틴다.
- 불필요한 출력/legacy/긴 캐시 생성 부담을 줄인다.
- SSH 없이 업데이트할 수 있게 만든다.
- 자동매매를 켜기 전에 상태 확인이 빠르고 정확하게 보이게 한다.

## 7. 서버/SSH 작업 현황

SSH가 잘 끊기고 명령어가 느렸던 원인 후보:
- 서버 메모리 1GB급으로 여유가 적음
- 봇 프로세스가 CPU/메모리를 크게 사용
- journal 로그가 약 7GB 이상 쌓여 있었음
- Swap이 없었음
- systemd timer/daemon-reload가 timeout을 만들었음

완료한 서버 작업:
- journal 로그 정리
- 디스크 사용률 일부 개선
- 512MiB Swap 생성
- systemd timer 자동배포 방식 폐기
- 10분 cron 자동배포 제거
- 서버 스크립트 유지
  - `/usr/local/bin/tradingbot_auto_deploy.sh`
  - `/usr/local/bin/tradingbot_deploy_status.sh`
  - `/usr/local/bin/tradingbot_admin_check.sh`
  - `/usr/local/bin/tradingbot_restart.sh`
- sudoers 등록 완료

서버 스펙업은 보류한다.
- 비용 부담 때문에 현재 서버를 일단 유지한다.
- 대신 v2.6.25부터 봇 다이어트/출력직원 경량화를 우선한다.

## 9. 운영 원칙 고정

- 문제가 생기면 조건 조이기보다 원인을 먼저 찾는다.
- 실거래/눌림/시장자료 직원은 멈추지 않게 한다.
- 급등복구/복기/긴 출력은 보조 작업이며 본선을 막으면 대기시킨다.
- 자동매매 OFF는 신규매수만 끄는 것이고 기존 보유 관리는 계속 책임진다.
- 수익보호 기준은 실거래 성과만 사용하고 paper 성과와 섞지 않는다.
- 눌림대기/PENDING/관심 후보는 푸시하지 않고 명령어 화면에서만 본다.
- 실제 푸시는 매수/매도 판단과 직접 연결된 것만 보낸다.
